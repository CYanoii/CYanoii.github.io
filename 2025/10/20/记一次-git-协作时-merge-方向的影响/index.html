记一次 git 协作时 merge 方向的影响

发表于 2025-10-20 分类于 学习 Valine：
本文字数： 1.5k 阅读时长 ≈ 3 分钟
在一次项目 git 协作中，由于 merge 方向不同产生了两张拓扑相同但展示结果不同的情况。

产生问题的流程

阶段一

  • 时间点 1：存在 origin/master(1)（括号内容表示该指针当前指向节点 1），我开了它的的本地分支 master(1)，并继续由此开了本地分支 dev-C(1)，在其上开发；
  • 时间点 2：origin/master(1) 被 push 一次提交，前进为 origin/master(2)；
  • 时间点 3 4 5：我在 dev-C(1) 做开发，commit 3 次，得到 dev-C(5)；
  • 时间点 6：我在本地 merge dev-C(5) to master(1) 得到 merge 节点 master(6)，之后 push 到远程。

  • 时间点 7：Z 在 dev-Z(2) 上 commit 1 次，推进到 dev-Z(7)，之后在 master(2) 上 pull origin/master(6)，准备合并。

观察

当时我感觉反顺序的 merge 会导致实际结果错误，但观察后发现两种操作后图的拓扑形状其实是一致的，反顺序是会导致显示逻辑错误。这是由于在反顺序操作中，master 才是被合并的一方，因而在显示上会成为一条颜色相同的分支：

总结来说，在需要来回 merge 的情况下，先后顺序不改变实际结果，但打乱显示逻辑。

后续实验

Pull request 和 merge 的显示逻辑区别

我之前认为，Pull request(PR) 和 merge 没有本质区别，即 PR 只是需要先在网站上审查再执行 merge，但实际实验发现并不是。在创建了开发分支，但主分支没有前进时进行 merge 开发分支 to 主分支 时（换句话说，逻辑上创建了分支但实际上提交树还没有真正分叉时），在显示逻辑上就不会产生 merge 节点，而显示为直接在主分支上做的开发。如：
