博客组织与文章书写总则

发表于 2024-10-09 更新于 2025-01-03 分类于 规则 Valine：
本文字数： 2.7k 阅读时长 ≈ 4 分钟
介绍了自己总结的博客组织与文章书写的重点和难点，同时规定了二者的具体要求。

前言

由于自己有一点强迫症和完美主义，于是一直想找到“完美”的博客组织方案和统一的文章书写方案。这两者我花过许多的时间去构思，也做过很多实践，比如说我大改过博客的组织逻辑，更改了几乎所有文章标签和分类；我也写过很多版的书写规定文档，尝试着明确规定许多书写细节。现在我想把这些我以往长期思考的东西全部整理到一篇博客文章上以便于自己随时查阅和修改，于是便有了这篇文章。

在前言的余下内容中，我会分别列举二者在方案制定上我总结的重点和难点。之后在下一章我会列出我所制定的博客组织与文章书写总则，同时这部分内容本身也会依照其规定实现动态更新与记录（博客的所有内容会加以实践并动态同步）。

关于博客组织方案：重点有两处，一个是分类，一个是更新。

首先介绍分类。一般博客框架会提供两种逻辑来对文章进行分类，一个叫“类别”，一个叫“标签”。类别与文章的关系是一对多的，即一篇文章只能从属于一个类别；标签与文章的关系是多对多的，即一篇文章可以有多个标签。因此在使用类别时，需要对所有文章的内容关系做完全划分，就比如将地球划分为陆地和海洋；在使用标签时，则要尽可能的多而全，以完全概况文章的内容。此时则分别引出二者的难点。对于分类，事实上很多时候我们很难做出完全划分，想要达到完全划分过于细致则会导致内容相互交融，过于宽泛又会导致分类没啥意义；对于标签，同样内容用不同表达描述而打成不同标签的情况屡见不鲜，且使用尽可能多的标签以更全面的概括文章本身就是一个很主观的事情。

随后介绍更新。首先文章的更新是必要的。从笔误的角度出发，错误需要被纠正；从文章时效性的角度出发，以技术科普为例，随着技术的迭代更新，文章的内容可能会过时或者出现错误，内容需要被纠正；从连载的角度出发，内容需要被更新。而文章更新的难点就是如何在更新的同时记录下统一且详细的更新日志。（2025.1.3 我认为更新日志作用有限且过于繁琐，因此更新时不再做记录）

关于文章书写方案：文章书写的重点与难点类似，因此这里统一算“痛点”吧。主要痛点一个是制定的标准往往不够具体，得到了实际使用时才发现完全没有做规定；另一个是在不同的位置、程序中，文字的排版与展现形式都大有不同，有些地方还会有特殊要求，因此为了在不同的地方都得到合理且舒适的书写与观看体验，不得不针对具体的书写场景制定具有针对性的文章书写方案。

博客组织方案

分类

  • 分为“规则”、“日常”、“学习”、“合集”。
    • 规则：管理组织博客的文章，比如这篇。
    • 日常：书写时主观上日常的文章，即使其中有学习的内容。
    • 学习：书写时主观上学习的文章，即使其中有日常的内容。
    • 合集：为了方便，根据某些规律而总结的多种内容的集合。
  • 对于前面提到的“主观”，指书写文章的目的和书写时所抱有的心态。如吐槽性质的竞赛比赛回忆就应该归类为日常，尽管它里面会提到一些竞赛的专业知识；而游戏攻略就应该归类为学习，因为书写时是抱着严肃认真的态度去书写心得或总结。总之“日常”和“学习”分类与内容类型无关，只与书写目的和心态有关。

标签

  • 尽可能的细化标签。
  • 尽可能概括文章的所有内容。
  • 严格检查以禁止同样内容用不同表达描述而打成不同标签的情况发生。

更新

  • 为了避免频繁的更新，文章以草稿格式（draft）新建，完成初版后才能够发布为文章（post）。
  • “规则”分类下的文章要慎重更新。一旦进行了更新，要对其造成了影响的文章进行更新。
  • 若存在大量错误内容或需要更新的内容时（自己度量），则放弃原文章，在其最前方追加“废弃提醒”栏。其格式如下：
    =废弃提醒内容开始=
    一级标题：废弃提醒
    本文章因 XXX 原因现已废弃，其中内容请酌情参考。（可追加一句“本文章的新地址请见 XXX。”）
    =废弃提醒内容结束=

（2025.1.3 更新章节下述内容作废）

  • 禁止发布未完成或更新中的文章。
  • 文章发布后的改动才记为更新。
  • 再小的更新也需要记录，即使是仅修改了一个标点也需要记录。
  • “规则”分类下的文章要慎重更新。一旦进行了更新，要对其造成了影响的文章进行更新。
  • 在存在更新行为的文章最后追加“更新日志”栏。其格式如下：
    =更新日志内容开始=
    一级标题：更新日志
    2024/10/03
    详细说明本次更新将什么更新为了什么。可多行。
    2024/10/02
    详细说明本次更新将什么更新为了什么。可多行。
    =更新日志内容结束=
  • 若存在大量需要更新的内容时（自己度量），则放弃原文章，在其最前方追加“废弃提醒”栏。其格式如下：
    =废弃提醒内容开始=
    一级标题：废弃提醒
    本文章因 XXX 原因而需要进行大量更新，现已废弃，其中内容请酌情参考。（可追加一句“本文章的新地址请见 XXX。”）
    =废弃提醒内容结束=
  • 追加“更新日志”栏或“废弃提醒”栏本身不算做更新。

文章书写方案

Markdown

  • 加粗/倾斜的使用：
    • 在对部分进行加粗/倾斜时，不需要囊括句子的标点。
    • 在对整个句子进行加粗/倾斜时，需要囊括句子的标点。
    • 在对带有冒号的部分进行加粗/倾斜时，需要囊括冒号。
    • 在对带有引号的部分进行加粗/倾斜时，需要囊括引号。
  • 空格的使用：
    • 中文和中文标点符号间不需要空格。
    • 英文和中文标点符号间不需要空格。
    • 中文和英文间需要一个空格。
    • 除符号外任何内容与行内公式间需要一个空格。
    • 除符号外任何内容与代码格式间需要一个空格。
    • 英文和英文标点符号间按照英文书写需求使用空格。
  • 对于中文为主体的文章，一律使用中文符号。
  • 对于英文为主体的文章，一律使用英文符号。

Word

  • 加粗/倾斜的使用：
    • 在对部分进行加粗/倾斜时，不需要囊括句子的标点。
    • 在对整个句子进行加粗/倾斜时，需要囊括句子的标点。
    • 在对带有冒号的部分进行加粗/倾斜时，需要囊括冒号。
    • 在对带有引号的部分进行加粗/倾斜时，需要囊括引号。
  • 空格的使用：
    • 中文和中文标点符号间不需要空格。
    • 英文和中文标点符号间不需要空格。
    • 中文和英文间不需要空格。
    • 除符号外任何内容与行内公式间需要一个空格。
    • 英文和英文标点符号间按照英文书写需求使用空格。
  • 对于中文为主体的文章，一律使用中文符号。
  • 对于英文为主体的文章，一律使用英文符号。

Code

  • 除了英文间，不需要空格。
  • 符号统一，全使用中文符号或英文符号，注意各自的空格规则。

Windows

  • 文件命名时尽可能不使用空格，某些情况下可以使用“_”代替空格。
  • 项目内文件命名时优先遵循该项目指定的命名规则。
  • 符号统一，全使用中文符号或英文符号，注意各自的空格规则。
